# ADR-007: Expand Vitest Stack with React Testing Library and jsdom

## Status

Accepted

## Date

2026-05-16

## Context

The project's `vitest.config.ts` runs in the `node` environment and covers `src/lib/**` + `app/api/**/route.ts` with 60% line / 50% branch thresholds. The character sheet rework introduces non-trivial React components: `SheetRenderer` orchestrates `react-hook-form` + dynamic Zod resolver + live derivation; `BodyMap` redesigns a 12-region interactive SVG whose click behaviour affects sheet state. Bugs in either are easy to introduce and hard to catch in a CI without UI assertions. The PRD explicitly states "Sem testes automatizados nesta fase" for the original MVP, but the rework reaches a complexity threshold where component smoke tests are cheaper than the regressions they prevent.

## Decision

Add `@testing-library/react`, `@testing-library/jest-dom`, `@testing-library/user-event`, and `jsdom` as devDependencies. Update `vitest.config.ts` to support a second project (or `environmentMatchGlobs`) so that:

- Files under `tests/lib/**`, `tests/api/**` keep running in `node`.
- Files under `tests/components/**` run in `jsdom`.

Component tests are limited to **smoke tests** for `SheetRenderer` (renders a known schema, autosave fires once after debounce, derived field updates on base change) and `BodyMap` (renders 12 regions, clicking a region opens the dialog, severity badge applies the expected colour class). They are not exhaustive UI tests and intentionally avoid pixel-level snapshots.

## Alternatives Considered

### Alternative 1: Lib + API only, no component tests

- **Description**: Stay in `node` env; cover the pure logic (`src/lib/sheets/*`) and HTTP contracts.
- **Pros**: Zero infra change.
- **Cons**: Misses the integration risk between RHF, the dynamic Zod resolver, and the live derivation â€” the highest-risk surface of the feature.
- **Why rejected**: The bugs we expect (renderer wiring, autosave timing, formula scope) all live in components.

### Alternative 2: Add Playwright for end-to-end coverage

- **Description**: Spin up Playwright; test the live `npm run dev` server flows.
- **Pros**: Highest-fidelity assertions, exercises the full stack.
- **Cons**: Heavier infra (browser binaries in CI); much slower; overkill when the unit-of-risk is the renderer in isolation.
- **Why rejected**: Cost/benefit favours RTL for this feature; Playwright can come later if smoke E2E becomes a priority.

## Consequences

### Positive

- The new RHF + dynamic Zod + derivation pipeline gets at least one assertion before it ships.
- BodyMap interactivity regressions surface in CI rather than in the editor.
- The test stack matches Next.js community norms (RTL + jsdom + vitest).

### Negative

- ~50MB of new devDependencies.
- One `vitest.config.ts` change to support split environments.
- A small ongoing burden: smoke tests must be kept in sync with renderer prop shapes.

### Risks

- Test flakiness from RHF's debounced state and `useEffect` chains â€” mitigated by `await waitFor(...)` patterns and `vi.useFakeTimers()` for the autosave timing tests.

## Implementation Notes

- `vitest.config.ts` gets `test.projects` (or `environmentMatchGlobs`) splitting `tests/components/**` to `jsdom`.
- `tests/setup-dom.ts` extends `expect` with `@testing-library/jest-dom` matchers.
- Component tests live at `tests/components/SheetRenderer.test.tsx` and `tests/components/BodyMap.test.tsx`.
- Coverage threshold stays at 60%/50% scoped to `src/lib/**` and `app/api/**/route.ts`; component tests are gating quality but not coverage targets.
