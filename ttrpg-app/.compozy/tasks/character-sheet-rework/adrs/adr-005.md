# ADR-005: Isomorphic Derivation Library with Server-Canonical GET

## Status

Accepted

## Date

2026-05-16

## Context

Effective sheet values are a function of three inputs: the `dataJson` base values, the active `CharacterCondition[]` (including their `modifiersJson` overrides plus the `injuryPresets` from the system schema), and the derived-field formulas. The user needs live preview: as they type in a base field or attach a condition, derived fields update without round-trips. The server also needs to expose effective values for read paths (RSC pages, future export, future API consumers) and as a canonical reference for the client on initial load.

## Decision

Place the derivation logic in pure, isomorphic modules under `src/lib/sheets/`:

- `parser.ts` — Zod schema for the JSON sheet schema; produces typed `SheetSchema`.
- `formula.ts` — recursive-descent parser/evaluator (ADR-002).
- `applyModifiers.ts` — merges condition presets and `modifiersJson` overrides into a `{ value, breakdown[] }` per affected field.
- `derive.ts` — orchestrates `applyModifiers` → formula evaluation → returns `EffectiveValues` and a `Breakdown` map keyed by field id for tooltips.

These modules have no React, no fetch, no Prisma. Both the server (in `GET /api/characters/[projectId]/[characterId]/sheet`) and the client (in `SheetRenderer` via `useMemo` over `watch()` output and the current `conditions` array) call the same `derive(baseValues, conditions, schema)` function. The server returns `{ schema, base, effective, breakdown }` so the initial render matches; subsequent edits recalculate locally until the user navigates away.

## Alternatives Considered

### Alternative 1: Client-only derivation

- **Description**: Server returns only `{ schema, base, conditions }`; client computes effective values.
- **Pros**: Smaller GET response; no server CPU on read.
- **Cons**: SSR/RSC paths that render the sheet (e.g., a future read-only export view) need to ship the eval bundle and run it during server render anyway; no canonical source if a non-React consumer arrives.
- **Why rejected**: Same code path will exist on the server eventually; centralising avoids future divergence.

### Alternative 2: Server-recalculate per PATCH round-trip

- **Description**: Each autosave PATCH returns the recomputed effective snapshot; client doesn't compute locally.
- **Pros**: Server is the single source of truth.
- **Cons**: 800ms debounce + network turns derived-field UI into laggy ghosts; user types and waits to see the new effective value.
- **Why rejected**: Bad UX given the debounce period; not justified when the function is deterministic.

## Consequences

### Positive

- One implementation, one set of tests; no risk of server/client formulas drifting.
- Live UI feedback under autosave debounce.
- Pure functions are trivial to unit-test (no React, no DB).
- Reusable for future read paths (PDF export, embed views).

### Negative

- Eval bundle ships to the browser (small — ADR-002 keeps it tiny).
- A client must trust its own computation between saves; if the schema changes mid-session via a hot-reload, the client's view diverges until refresh — acceptable.

### Risks

- A divergence between client and server logic introduced by accidental React-only imports — mitigated by an ESLint rule (or, at minimum, a directory README) forbidding React/Next imports inside `src/lib/sheets/`.

## Implementation Notes

- Type contract: `derive(base: Record<string, FieldValue>, conditions: CharacterCondition[], schema: SheetSchema): { effective: Record<string, FieldValue>, breakdown: Record<string, BreakdownEntry[]> }`.
- `BreakdownEntry`: `{ source: "preset" | "override" | "formula"; delta?: number; reason?: string; conditionId?: string }`.
- `SheetRenderer` consumes `breakdown[fieldId]` to render the tooltip on derived/affected fields.
