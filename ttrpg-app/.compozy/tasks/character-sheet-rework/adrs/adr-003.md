# ADR-003: Adopt React Hook Form for Dynamic Sheet Rendering

## Status

Accepted

## Date

2026-05-16

## Context

Existing forms in the app (`src/components/characters/CharactersClient.tsx`, `CharacterDetailClient.tsx`, `GlossaryDetailClient.tsx`) are vanilla controlled inputs backed by `useState`. They have 3–10 fields, no nested arrays, and no autosave per field. The character sheet, by contrast, will render dozens of fields per system (D&D 5e has 6 attribute scores, 18 skills with checkboxes, repeating attack/equipment/spell lists, plus textareas), with debounced field-level autosave and live recomputation of derived values. A single top-level `useState` re-renders every field on every keystroke; manual `useReducer` plumbing for nested repeating lists is error-prone and would need to reimplement standard patterns (array helpers, dirty tracking, validation integration).

## Decision

Add `react-hook-form` (~25KB) and `@hookform/resolvers` as the form layer for `SheetRenderer`. Use `useForm` with `defaultValues` derived from the sheet's `baseValues`, `Controller` for non-native inputs (Radix/ShadCN), `useFieldArray` for `repeating-list` fields, and `watch`/subscriptions to feed the derivation engine without forcing whole-tree re-renders. Other existing forms in the codebase remain on `useState` — adoption is scoped to the sheet feature for now.

## Alternatives Considered

### Alternative 1: Stay on `useState` / `useReducer`

- **Description**: Maintain the current project pattern; centralise state in one reducer.
- **Pros**: No new dependency; perfectly consistent with the rest of the app.
- **Cons**: Re-renders the entire sheet on every keystroke (perceptible at the upper bound of sheet size); `useFieldArray`-equivalent helpers must be hand-written; integrating Zod validation requires custom plumbing; debounced field-level dirty tracking is non-trivial.
- **Why rejected**: The custom code needed to make this performant approaches the size of an RHF integration, without the ecosystem benefits.

### Alternative 2: Custom `useSheetForm` hook

- **Description**: Build a small purpose-built hook using `useReducer` + `useSyncExternalStore` for field-level subscriptions.
- **Pros**: No dependency; tailored to the sheet's shape.
- **Cons**: Reinvents RHF's primary value proposition; ongoing maintenance burden; no community knowledge to draw on.
- **Why rejected**: YAGNI in reverse — this is exactly the problem RHF solves, and the project will outlive whoever writes the custom hook.

## Consequences

### Positive

- Field-level re-renders keep the editor responsive at scale.
- `useFieldArray` covers the repeating-list field type out of the box.
- `@hookform/resolvers` integrates cleanly with the dynamic Zod validator from ADR-006.
- Mature, well-typed, widely understood.

### Negative

- Introduces the first form library to the project; some inconsistency with existing forms.
- New contributors need to know RHF to touch the sheet.

### Risks

- `Controller` boilerplate for Radix-based inputs can leak verbosity into field components — mitigated by writing a thin `<SheetField>` wrapper per field type that hides the `Controller` glue.

## Implementation Notes

- Adopted versions: `react-hook-form@^7`, `@hookform/resolvers@^3`.
- Pattern: each `SheetField`-type component (`TextField`, `NumberField`, `SelectField`, …) receives `name` and `control` props and uses `Controller`. The `RepeatingListField` uses `useFieldArray`.
- `SheetRenderer` calls `useForm({ defaultValues: baseValues, resolver: zodResolver(dynamicSchema) })` and feeds `watch()` into a memoised `applyModifiers` + formula evaluator for live derived values.
