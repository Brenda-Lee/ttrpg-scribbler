# ADR-004: DB-First Sheet Schema with Bundled JSON Fallback

## Status

Accepted

## Date

2026-05-16

## Context

The sheet schema for each `System` (D&D 5e, Tormenta 20, Generic) must be available at runtime to render forms and validate inputs. Two natural sources exist: (a) `System.rulesJson` populated by the seed in the database; (b) JSON files committed to the repo at `prisma/sheets/*.json`. The PRD picks the DB as the canonical store, but the DB row can be missing, malformed, or out-of-date relative to the code that consumes it — particularly during development, after schema iterations, or when running fresh against an unseeded database.

## Decision

The runtime resolver in `src/lib/sheets/loadSchema.ts` reads `System.rulesJson` via Prisma first. If the value is `null`, fails Zod validation, or carries a `schemaVersion` older than the bundled one for that `systemSlug`, fall back to the JSON file imported from `prisma/sheets/{slug}.json`. Both sources pass through the same Zod parser (`src/lib/sheets/parser.ts`) so the consumer sees a single typed `SheetSchema`. The resolver caches per `(systemSlug, schemaVersion)` in module scope. Seed writes the same bundled JSON to `System.rulesJson` so DB and code stay aligned in normal flows.

## Alternatives Considered

### Alternative 1: DB only

- **Description**: Always read `System.rulesJson`; error if missing.
- **Pros**: One source of truth at runtime.
- **Cons**: Requires running `db:seed` after every schema change; awkward in CI, in fresh clones, and during code review of schema diffs.
- **Why rejected**: Makes the schema invisible to TypeScript at build time and degrades dev ergonomics.

### Alternative 2: Bundled only

- **Description**: Always import JSON from disk; ignore `System.rulesJson`.
- **Pros**: Zero round-trip; type-safe at boot.
- **Cons**: Forecloses the post-MVP "DM customizes their system" workflow (PRD Fases futuras) and contradicts the PRD's stated source of truth.
- **Why rejected**: Locks out the future override pathway the product explicitly anticipates.

## Consequences

### Positive

- Fresh clones work without seeding (UI still renders sheets).
- Production can override defaults by writing to `System.rulesJson`.
- The Zod parser keeps a single shape contract regardless of source.
- Future "DM-customised sheets" plug into the existing path with no rework.

### Negative

- Two sources can diverge if a DB row is hand-edited and the bundled file is updated — `schemaVersion` comparison plus a startup log line keeps this visible.
- Slightly more code in the resolver (fallback branch).

### Risks

- A buggy `System.rulesJson` could silently fall back without operator awareness — mitigated by emitting a console warning with the validation issue summary on every fallback.

## Implementation Notes

- `src/lib/sheets/catalog/{dnd5e,tormenta20,generic}.ts` re-exports the typed schema parsed from the JSON at module load (so TypeScript can verify shape via Zod's inferred type).
- `prisma/sheets/{dnd5e,tormenta20,generic}.json` is the authored source; the TS catalog modules `import` it with `assert { type: "json" }` (or via `tsx` JSON import in the seed).
- Resolver signature: `loadSheetSchema(systemSlug: string | null): Promise<SheetSchema>` returning the generic schema when `systemSlug` is null.
