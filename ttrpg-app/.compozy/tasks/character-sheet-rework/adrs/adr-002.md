# ADR-002: Custom Recursive-Descent Formula Evaluator (Zero Dependencies)

## Status

Accepted

## Date

2026-05-16

## Context

`derived` fields in the sheet schema (e.g., `"defesa_efetiva": "defesa_base + mod(des)"`) need a deterministic, side-effect-free expression evaluator that runs identically on server and client. The grammar required is small: numeric literals, field references, `+ - * /`, parentheses, and a fixed set of functions (`floor`, `min`, `max`, `mod` — where `mod(attr) = floor((attr - 10) / 2)`). Untrusted input is the schema itself (authored by the project owner via JSON) plus user values; an evaluator that supports too much (e.g., `mathjs` global symbol table) is a needless attack surface.

## Decision

Implement a tokenizer + recursive-descent parser in `src/lib/sheets/formula.ts` (~150 LOC), exposing `parse(expr: string): FormulaNode` and `evaluate(node: FormulaNode, scope: Record<string, number>): number`. Grammar whitelist: numeric literals, identifiers (resolved against `scope`), parenthesised expressions, binary `+ - * /` with standard precedence, and a fixed function table. Parsing happens once per `schemaVersion` and the resulting AST is cached. Division by zero yields `NaN` and is surfaced in the derivation breakdown rather than thrown.

## Alternatives Considered

### Alternative 1: `expr-eval` library

- **Description**: ~10KB JS library with a `Parser` class.
- **Pros**: Off-the-shelf, well-tested.
- **Cons**: Ships features we must explicitly disable (assignments, comparison operators, conditional expressions); still requires a sandboxing wrapper.
- **Why rejected**: The wrapping work approaches the cost of writing a focused parser, with a runtime dependency we don't control.

### Alternative 2: `mathjs`

- **Description**: Full math library with `evaluate(expr, scope)`.
- **Pros**: Extremely powerful, supports complex math.
- **Cons**: ~150KB minified; supports far more than we need; sandboxing requires careful `import` filtering.
- **Why rejected**: Over-engineered for the four operators plus three functions we actually use.

### Alternative 3: Pure JSON AST in the schema (no string parser)

- **Description**: Schema declares `{ op: "add", args: [{ ref: "defesa_base" }, { fn: "mod", args: [{ ref: "des" }] }] }`.
- **Pros**: No parser needed; trivially Zod-validatable; impossible to inject.
- **Cons**: Schema authoring (PRs to `prisma/sheets/*.json`) becomes verbose; harder for humans to write and review.
- **Why rejected**: Hurts the primary authoring workflow. Kept as a possible escape hatch if expression complexity ever exceeds the parser's grammar.

## Consequences

### Positive

- Zero new runtime dependencies; the project keeps a tight `package.json`.
- Bundle impact on client is minimal (parser is small and tree-shakable).
- Identical code path on server and client (isomorphic — ADR-005).
- Easy to extend the function table as needs grow.

### Negative

- ~150 LOC of parser code to maintain.
- Must hand-write parser tests covering precedence, parenthesisation, errors, divisions, and unknown-identifier handling.

### Risks

- Unsupported syntax in a hand-authored schema fails at runtime — mitigated by a `parser.ts`-level Zod validator that parses every `formula` string at schema load time and surfaces errors before the sheet is served.

## Implementation Notes

- Recommended file layout:
  - `src/lib/sheets/formula.ts` — `tokenize`, `parse`, `evaluate`, types.
  - `tests/lib/formula.test.ts` — table-driven tests covering grammar.
- Cache compiled ASTs in a `WeakMap<SheetSchema, Map<fieldId, FormulaNode>>` so repeated GETs don't reparse.
- On unknown identifier, evaluator returns `NaN` and pushes a `BreakdownEntry { field, reason: "missing_ref" }` so the UI can surface the issue.
