# ADR-006: Dynamically-Generated Zod Validator for Sheet PATCH

## Status

Accepted

## Date

2026-05-16

## Context

`PATCH /api/characters/[projectId]/[characterId]/sheet` accepts partial updates to a `dataJson` whose shape varies per system. Static Zod schemas in the route file can't model the shape — there is no compile-time list of fields. The server still needs to reject malformed types (e.g., a string sent where a number field is expected), unknown keys (to prevent `dataJson` pollution), and structurally invalid payloads (prototype-pollution-style keys, nested objects where primitives belong).

## Decision

Generate a Zod schema at request time from the active `SheetSchema`. A helper `src/lib/sheets/zodFromSchema.ts` exports `buildPatchSchema(sheetSchema: SheetSchema): z.ZodType<SheetPatch>` that walks the schema sections/fields and produces a `z.object({ [fieldId]: z.string().optional() | z.number().optional() | ... }).strict()` (strict mode rejects unknown keys). `repeating-list` becomes `z.array(z.object({ ... }))`. The generated schema is cached per `(systemSlug, schemaVersion)`. The route loads the schema via the resolver from ADR-004, builds the Zod schema (cache-hit on warm requests), and validates the request body.

## Alternatives Considered

### Alternative 1: `z.record(z.unknown())`

- **Description**: Accept any JSON object; trust the client.
- **Pros**: Trivial to implement.
- **Cons**: A buggy or malicious client can write arbitrary keys to `dataJson` (e.g., `__proto__`, garbage fields) that the renderer ignores but DB rows accumulate; type mismatches surface only at render time.
- **Why rejected**: Forfeits server-side integrity for marginal simplicity gain.

### Alternative 2: Hybrid — fixed Zod shape + post-parse filter

- **Description**: Use `z.record(z.union([z.string(), z.number(), z.boolean(), z.array(z.unknown())]))`, then filter keys to those present in `SheetSchema`.
- **Pros**: No dynamic schema generation.
- **Cons**: Drops type-level mismatches (e.g., string in a number field passes the record check); the filter is essentially half of the work `zodFromSchema` does anyway.
- **Why rejected**: Catches fewer mistakes for similar code volume.

## Consequences

### Positive

- Type errors are caught at the API boundary with usable Zod issue messages.
- `.strict()` blocks key pollution and accidental key drift.
- The same `zodFromSchema` powers form resolver on the client (ADR-003).

### Negative

- A small CPU cost per cold cache to build the Zod schema.
- One more pure module to test.

### Risks

- A schema with a field type the generator doesn't yet support would throw at build time — the generator's own Zod validator (in `parser.ts`) restricts allowed field types to the supported set.

## Implementation Notes

- Cache key: `${systemSlug}:${schemaVersion}`.
- Module exports both `buildPatchSchema` (server) and `buildResolverSchema` (client) — they may diverge later (e.g., the client tolerates partials more loosely during typing); for MVP they can be the same function.
- Repeated-list items use a nested `z.object({ ... }).strict()` to keep the no-unknown-keys guarantee at every level.
