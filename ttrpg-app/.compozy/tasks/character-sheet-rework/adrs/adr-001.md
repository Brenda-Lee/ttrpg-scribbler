# ADR-001: Separate `CharacterSheet` Entity for System-Specific Game Data

## Status

Accepted

## Date

2026-05-16

## Context

The current `Character` model in `prisma/schema.prisma:194-210` mixes narrative metadata (name, role, bio, avatarAssetId) with what would soon become a system-dependent payload (`attributesJson` — a free-form JSON string used inconsistently in the seed). The PRD introduces a structured per-system character sheet (D&D 5e, Tormenta 20, Generic) with potentially dozens of fields, sections, repeating lists, and derived values. Coexisting that payload alongside `Character` would:

- Mix two access patterns (catalog browsing of characters vs. heavy form editing of one sheet) on a single row.
- Force every `Character.findMany` to hydrate large JSON blobs even when listing.
- Conflate "world links" (`attributesJson` per PRD: keeps as map of related items/locations) with "game state".
- Make versioning the sheet shape harder (no place to attach `schemaVersion`).

## Decision

Introduce a new `CharacterSheet` model in `prisma/schema.prisma` with a 1:1 relation to `Character` (`@unique characterId`, `onDelete: Cascade`). It carries `systemSlug`, `schemaVersion`, `dataJson` (base values only — derived values are computed at read time), and timestamps. `Character.bio` (relabeled "Resumo" in the UI) and `Character.attributesJson` (world links) stay on `Character` unchanged. Sheet creation happens automatically inside the same transaction as character creation in `POST /api/characters/[projectId]`, using `Project.system.rulesJson` (or `generic` fallback) to seed defaults.

## Alternatives Considered

### Alternative 1: Single `Character.sheetJson` column

- **Description**: Add `Character.sheetJson String?` carrying the entire sheet payload.
- **Pros**: One migration; one query path; no relation overhead.
- **Cons**: Bloats every character listing; mixes catalog and game data; no natural place for `schemaVersion`.
- **Why rejected**: List endpoints (`/characters` page) would carry the heavy JSON unnecessarily; conceptual coupling.

### Alternative 2: Reuse `Character.attributesJson`

- **Description**: Move the sheet into the existing `attributesJson` slot.
- **Pros**: Zero schema changes.
- **Cons**: Loses the semantic split between "world links" and "game state" that the PRD goal #4/#5 explicitly preserves; existing seed data already uses `attributesJson` for non-sheet content.
- **Why rejected**: Conflicts with the PRD's explicit separation; would require reinterpreting existing fixture data.

## Consequences

### Positive

- `Character` queries stay lean; sheet hydration is opt-in.
- `schemaVersion` lives where it belongs and supports future migrations.
- Cascade delete keeps cleanup trivial.
- The 1:1 contract is enforceable via `@unique`.

### Negative

- Two-table writes on character creation (mitigated by `prisma.$transaction`).
- A backfill step is needed for any existing characters in dev databases.

### Risks

- Drift between `Character` and `CharacterSheet` if a future code path creates `Character` directly without sheet — mitigated by funnelling all creation through the API route and seed helper.

## Implementation Notes

- New model fields: `id`, `characterId @unique`, `systemSlug`, `schemaVersion @default(1)`, `dataJson`, `createdAt`, `updatedAt`.
- Inverse relation on `Character`: `sheet CharacterSheet?`.
- Seed `prisma/seed.ts` extends to create a `CharacterSheet` for every demo character (idempotent: skip when one already exists).
- API route `POST /api/characters/[projectId]/route.ts` wraps `prisma.character.create` + `prisma.characterSheet.create` in `$transaction`.
